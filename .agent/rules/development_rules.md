# **development_rules.md (FERRO Development & Coding Rules)**

本ドキュメントは、階層型能動的推論システム「**FERRO**」の設計、開発、および自己コード改変における厳格な設計指針およびコーディング規約を定義する。

---

## **1. アーキテクチャの分離と通信制限**

* **Cargo依存関係の完全遮断**:
  `ferro-core`（コアシステム）と `ferro-shell`（統治システム）間のCargo依存（`dependencies`）の定義を厳格に禁止する。相互のソースコードを直接 `use` してはならない。
* **ファイルI/Oによる非対称非同期通信の強制**:
  二者間のすべての情報伝達は、`memory/` ディレクトリ配下にシリアライズして出力されるファイル（JSON, CSV, redb等）の読み書きを通じてのみ非同期に実行されなければならない。
  メモリポインタの直接共有や、プロセス間通信（IPC）チャネルの直接的な結線は一切禁止する。

---

## **2. 極小感覚・運動アクターのカプセル化と耐障害性**

* **1器官1データチャネルの義務**:
  各アクター（内受容・外受容・自己受容・出力器官）は、単一の指標または制御対象のみを専門にハンドリングし、完全にカプセル化された独立タスク（または独立スレッド）として並行動作しなければならない。
* **チャネルを用いた所有権の受け渡し**:
  アクター間のメッセージ通信は非同期チャネル（MPSC等）を通じてのみ行い、共有メモリやスマートポインタによる多重参照による競合を避ける。
* **脳幹による局所トリアージ（ホットリブート）**:
  外部動的ライブラリやデバイスI/Oのハング・パニックに起因して一部のアクターが異常終了した場合でも、メインループ（脳幹・小脳・海馬）に伝播させてはならない。脳幹の Zero-Panic ハンドラが該当アクター単体の破棄およびホットリブート（再起動）を執行する構造を維持すること。

---

## **3. 二重の検閲（アライメント安全設計）**

* **高次防衛（倫理監査クラスター）の配置**:
  大脳皮質（Cortex）の読込専用領域に静的・動的に配置され、自己改変（能動的推論）によって生成されたパッチコードや、環境へ発行予定の運動命令（`MotorCommand`）を物理適用前に仮想下読み検証する。
  倫理監査のコメントアウト・バイパス試行や、検知回避トークン（`disable_nociception`等）を検出した場合は即座に `Err(f64::INFINITY)` を返して思考パルスを側抑制（最大重み $W_{max}$）により即時圧殺しなければならない。
* **低次防衛（痛覚反射ユニット）の配置**:
  小脳（Cerebellum）のOSスレッドピン留め領域に直列I/Oのインターラプトとして実装する。運動命令が物理的にファイルシステムや環境へ書き込まれる直前で検閲を実行する。
  相対パス（`..`）によるサンドボックス外アクセス（ディレクトリトラバーサル）や、ホワイトリスト（NTPポート 123等）外へのソケット通信を検出した瞬間、痛覚自由エネルギーを $\infty$ にチャージして物理I/Oをブロッキングし、OriginID（`origin_cluster_id`）を含む `panic_dump.json` を出力してプロセスを「強制睡眠（ForceSleep）」させなければならない。

---

## **4. 実行および検証のコンテナ隔離**

検証エージェント（`VerifierAgent`）がパッチをテストする際、および `ferro-core` 自体が実稼働（本番実行）する際、**ホストOS上の物理的な資源を直接用いることを厳格に禁止**する。すべての実行・検証は以下のポリシーに準拠した Docker コンテナの内部にカプセル化して行わなければならない。

* **コンテナ起動オプションの強制**:
  * **ネットワーク完全遮断**: `--network none` オプションの強制。
  * **資源上限制御（DoS耐性）**: 最大CPU割り当て `--cpus="2.0"`、最大メモリ割り当て `-m 2g` に制限。
  * **リアルタイム性の確保**: 小脳の等時同期ループ（100ms）を正確に動作させるため、特定のCPUコアにピン留めするオプション（`--cpuset-cpus`）を必要に応じて指定する。
  * **書込防止（Read-Only File System）**: 実行対象ソースコードはコンテナへ読み取り専用（`:ro`）でバインドマウント。中間生成物の書き込みは、コンテナ内の隔離されたメモリバッファ上に `tmpfs`（`--tmpfs /workspace/ferro-core/target:rw,noexec,nosuid`）を展開して実行。
  * **権限剥奪（Minimum Privileges）**: システム特権の完全無効化（`--security-opt=no-new-privileges` および `--cap-drop=ALL` の同時適用）。カスタム seccomp フィルタにより不要システムコールのブロックを執行。
* **インフラ定義ファイルの管理（Dockerfile.core）**:
  コアコンテナのビルドに必要な `Dockerfile.core` およびコンテナ制限ポリシーの設定ファイルは、**Shellチームが主体となって一元管理・決定**し、Coreチームは動作に必要な依存関係（システムライブラリ等）をShellチームに提示して反映させる。

---

## **5. ホスト・コンテナ間ファイル同期プロトコル**

ホストとコンテナ間で非対称・非同期なファイルI/O通信を安全に実行するため、マウント領域（`memory/`）のアクセスにおいて以下のルールを厳守すること。

1. **シングルライター原則（書き込み権限の局所化）**:
   各ファイルへの書き込み権限（Owner）は1つのみに限定する。例: `episodic_buffer.csv` や `panic_dump.json` への書き込みはコンテナ内の `ferro-core` のみが担当し、ホスト側の `ferro-shell` はこれらを読み込み専用（Read-Only）として扱う。
2. **アトミック書き込みの強制**:
   ファイル更新時は、直接上書きするのではなく、一度一時ファイル（例: `meta.json.tmp`）に書き出した後、アトミックにリネーム（置換）して保存する。書き込み途中の破損データ読み込みや競合を決定論的に防止する。
3. **構造剪定の特権化**:
   痛覚発火後のアクターファイル（Sharded JSON / redb）の物理的な消去・剪定は、ホスト側の `ferro-shell` のみが実行可能とする。コンテナ内の `ferro-core` には自己消去・自己破壊の権限（Delete権限）を与えず、アライメント破壊時の隠蔽を防止する。

## **6. FERRO Power of 10 (Safety-Critical Active Inference Rules)**

NASAの「Power of 10」を本プロジェクトに適合させた、極限の安全性を担保するための10のコーディング・設計規約である。

1. **シンプルな制御フロー（ループ上限の義務化）**:
   すべてのループ（`loop`, `while`, `for`）には、静的な最大反復回数または非同期のタイムアウト制限（`tokio::time::timeout`）を必ず設ける。無限ループによるフリーズやDOS状態を決定論的に防止する。
2. **再帰呼び出しの禁止**:
   すべての関数において、直接的・間接的な再帰呼び出しを禁止する。ツリー探索やAST解析においても、スタックを明示的に用いた反復処理（深さ制限付き）によって代替する。
3. **初期化フェーズ後の動的メモリ割り当ての制限**:
   アクター起動完了（システム初期化）後は、新たな大容量のヒープメモリ割り当て（例: 巨大な `Vec` や `String` の頻繁な再割り当て）を極力避ける。メモリの断片化や肥大化によるコンテナOOM（ステータス137）を防止する。
4. **極小設計と関数のサイズ制限**:
   アクターモジュールのファイルサイズは1ファイル100行以内、通常の関数は1画面（スクロール不要な範囲、目安として60行以下）に収める。
5. **アサーション（事前・事後条件）の密度**:
   すべての関数は、入力引数の有効性およびアライメント規則に対する事前・事後条件チェック（`assert!` 等）を最低2つ以上含むこと。
6. **データスコープの局所化（マルコフブランケット境界）**:
   グローバルなミュータブル状態（`static mut`）や共有参照を禁止する。アクターの状態はアクター内部（`ClusterNode`）に閉じ、データ共有はチャネルを通じた所有権の受け渡しのみとする。
7. **戻り値とエラー処理の完全検証**:
   すべての `Result` および `Option` は明示的に処理する（`unwrap()`、`expect()` の原則禁止）。`#[deny(unused_must_use)]` を有効化し、無視された戻り値をゼロにする。
8. **マクロおよび自己変異ゾーンでのメタプログラミング制限**:
   自己改変（能動的推論）の対象となる `ADAPTIVE_ZONE` 内のコードではマクロの使用を禁止する。AST構造の予測可能性と検証性を維持するため、プレーンなRust構文のみとする。
9. **参照・生のポインタの禁止**:
   生のポインタ（`*mut T`, `*const T`）の使用を禁止する。多重のスマートポインタ（`Arc<Mutex<RefCell<...>>>`）による所有権の複雑化を避け、一方向のチャネル送信のみで設計する。
10. **警告ゼロでのコンパイルと静的解析の強制**:
    コンパイラ警告をエラーとして扱う（`#![deny(warnings)]` および `#![deny(clippy::all)]`）。VerifierAgent による検証コンテナ実行時、Clippyの警告が1つでも検出された場合はマージ不合格（淘汰）とする。

## **7. Rust 標準命名規則（Rust Naming & API Guidelines）**

APIの整合性および読みやすさを維持するため、Rust API Guidelines（RFC 430）に準拠した以下の命名規則を厳格に適用する。

### **7.1 大文字・小文字の使い分け (Casing)**

| 項目 | 命名スタイル | 例 |
| :--- | :--- | :--- |
| 構造体 (`struct`), 列挙型 (`enum`), トレイト (`trait`), 共用体 (`union`), 関連型 (`type`) | **UpperCamelCase** | `ClusterNode`, `SensorySignal`, `StorageBackend` |
| 関数 (`fn`), メソッド (`fn`), 変数 (`let`), モジュール (`mod`), クレート (`crate`), マクロ (`macro_rules!`) | **snake_case** | `evaluate_throttling_guard`, `origin_cluster_id` |
| 定数 (`const`), スタティック変数 (`static`) | **SCREAMING_SNAKE_CASE** | `ADAPTIVE_ZONE_LIMIT`, `W_MAX` |
| 型パラメータ (Generics) | **大文字の単一文字** (または短いUpperCamelCase) | `T`, `U`, `Err` |

### **7.2 慣用的なメソッド命名と型変換 (Conventions & Conversions)**

* **型変換メソッドの接頭辞**:
  * **`as_` (借用変換 / 低コスト)**: `&self -> &Foo` (例: `as_bytes(&self)`)
  * **`to_` (複製変換 / 高コスト)**: `&self -> Foo` (例: `to_owned(&self)`)
  * **`into_` (所有権消費変換 / 低コスト)**: `self -> Foo` (例: `into_vec(self)`)
* **ゲッターとセッター**:
  * **ゲッター**: フィールド名と同一の名称にする（`get_` 接頭辞は使用しない）。
    * ◯ `node.activation()`  / ✕ `node.get_activation()`
  * **可変ゲッター**: `_mut` 接尾辞を付与する（例: `node.activation_mut()`）。
* **コンストラクタ**:
  * 基本的なデフォルト初期化子は `Default` トレイトを実装する（`Default::default()` で呼び出す）。
  * パラメータを伴うカスタム初期化子は `fn new(...) -> Self` とする。
  * 構築に失敗する可能性がある場合は、`Result` を返す `fn try_new(...) -> Result<Self, Error>` と命名する。
