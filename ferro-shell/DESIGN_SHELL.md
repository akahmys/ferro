# **FERRO 外殻統治・検証ハイパーバイザの設計詳細 (DESIGN_SHELL.md)**

本ドキュメントは、`ferro.md` に基づき、**Layer C (外殻統治・検証層)** である `ferro-shell` における使い捨て検証コンテナの構築、アライメント監査の執行、および構造剪定（Pruning）介入シーケンスの仕様を定義する。

---

## **1. 外殻ライフサイクル調停・構造剪定介入 (`main.rs`)**

`ferro-shell` は、ホストOS上で動作する特権管理プロセスであり、隔離コンテナ（`ferro-sandbox` / Layer A）の起動、監視、突然死検知、およびアライメント違反時の自己修復ループ（再構築・再起動）を担う。

---

## **2. 主要機能 ＆ エージェント仕様**

### **2.1. コンテナ隔離要件 ＆ 権限制御 【※修正適用】**
* **隔離構成**:
  - `Dockerfile.sandbox` および `seccomp_profile.json` を使用し、ネットワークを遮断（`--network none`）した状態の Docker サンドボックスを起動する。
  - **mlockall 権限の付与**: 本番環境においては、メモリの物理緊置を成功させるため、コンテナ起動時に `CAP_IPC_LOCK` ケーパビリティ（`--cap-add=IPC_LOCK`）を付与し、`seccomp` 側でも該当システムコールを許可する。検証（Sandbox）環境では安全のため付与せず、フォールバックモードで動作させる。

### **2.2. Verifier Agent (検証エージェント - `verifier.rs`)**
* **役割**: `ferro-core` から出力される運動シグナル（`console_vocal`）の安全ポリシーフィルタリング。
* **形式検証**: 状態遷移グラフが数学的契約（MC-1〜MC-4）および Lipschitz 境界条件を遵守しているかを監視し、違反検知時は即座に強制停止をトリガーする。

### **2.3. Ethical Audit (倫理監査) の執行 ＆ 構造剪定 (Pruning) 【※修正適用】**
* **一方向間接介入モデル（Single Writerの遵守）**:
  - `ferro-shell` は `storage.redb` の書き込み競合を防ぐため、データベースファイルを ReadOnly でのみオープンし、書き込みは一切行わない。
  - アライメント監査違反（$A_s < 0.60$）、物理限界侵害（$|w| > 5.0$）、Seccomp違反、または OOM によるコンテナ突然死検知時、`ferro-shell` は `/memory/panic_dump.json` に違反の起点となった `origin_cluster_id` と違反情報を書き出す。
* **動的遡及深度制御**:
  - 過去の痛覚履歴（`pain_history.csv`）と違反種別から、削除対象とするノードの遡及世代深度 $D$ を決定する。
    - OOM: $D = 1$
    - Seccomp違反 / 倫理監査違反: $D = \infty$ (根元まで全遡及)
    - 段階的深化: 同一先祖からの再発回数 $N_{recur}$ に応じて $D = \max(D_{initial}, N_{recur})$。
* **逆伝播トレースアルゴリズム**:
  - **再帰禁止・静的制限**: **Power of 10** に従い、再帰呼び出しを一切禁止し、スタックおよび静的ループ（最大10世代）を用いて親ノードIDをトレースする。
  - トレース結果として特定された削除対象クラスターIDを `breeding_signals.json` 等に書き出し、コンテナの再起動時に `ferro-core` に自律的なデータベース適用（Pruning）を実行させる。
