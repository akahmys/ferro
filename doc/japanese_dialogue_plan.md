# **FERRO 日本語対話能力獲得計画 (Japanese Dialogue Acquisition Plan)**

## **1. 全体方針とコンセプト (Core Concept)**

FERRO（Hierarchical Active Inference System）は、自由エネルギー原理（FEP）に基づく予測誤差（Surprise）最小化ダイナミクスを応用して自己組織化します。
日本語の対話能力をゼロから獲得させるにあたり、一方的なデータの流し込み（受動学習）だけでは「能動的推論（Active Inference）による感覚運動ループのクローズ」が不十分になります。

そこで本計画では、以下の二本柱により段階的に日本語対話能力を定着させます。
1. **多段階カリキュラム（Stage 1〜6）**: 単語から文法、一対一の問答、そして日常会話への多段階ステップ。
2. **双方向対話環境（模擬チューター ＆ 人間介入）**: コアの運動出力（`vocal_text`）を検知し、即座に「意味のある応答」を感覚入力（`auditory`/`visual`）にフィードバックする仕組み。

## **2. 段階切教育カリキュラム設計 (Curriculum Stages)**

各ステージは、コアの脳領域（皮質、中脳、小脳、海馬）に特定の自己組織化ダイナミクスを引き起こすよう設計されています。

| ステージ | 難易度・カテゴリ | 入力形式 (刺激) | 期待されるコアの反応 / 学習目標 | 動作する数理・脳ダイナミクス |
| :--- | :--- | :--- | :--- | :--- |
| **Stage 1** | **基礎名詞 (Nouns)** | 「いぬ」「ねこ」「りんご」＋感覚特徴量 | 単語の概念認識。<br>感覚と記号の結合。 | 皮質クラスターの創出と有糸分裂（Mitosis）。 |
| **Stage 2** | **二語文 (Two Words)** | 「いぬ はしる」「そら あおい」<br>（スペース区切り） | 主語＋動詞/形容詞の関係性の把握。 | 中脳でのマルチトークン分解と、関連コンセプト間のシナプス結合強化。 |
| **Stage 3** | **文法と助詞 (Grammar)** | 「いぬ が はしる」「りんご は あかい」 | 助詞（が・は・を）を介した文法構造の受容。 | 側抑制（Lateral Inhibition）による、機能語（助詞）アクターと内容語の分化。 |
| **Stage 4** | **一対一問答 (QA)** | 「これ は なに」＋画像特徴量 | 特徴量に対応する単語の自律的出力。<br>（例：「いぬ」「りんご」を出力） | **運動出力の発火**。<br>小脳の痛覚反射・中脳の随伴発射相殺（Efference Cancellation）の連携。 |
| **Stage 5** | **日常会話 (Dialogue)** | 「こんにちは」「おはよ う」「げんき」 | 状況や挨拶に応じた、簡単な対話応答。<br>（例：「こんにちは」「げんき です」） | 状況コンセプト（コンテキスト）アクターを介した、複数概念の連想活性化。 |
| **Stage 6** | **大規模対話受容 (Ingestion)** | 大規模語彙データセット (5,000語〜) | 大規模対話データの受容処理による語彙概念数急増とKVSマイグレーションの検証。 | 5,000ノード到達時の `redb` 自動マイグレーション機構の作動。 |
| **Stage 7** | **実対話 (Interactive)** | 模擬チューター / 人間の入力 | 人間やAI教師とのリアルタイム双方向対話。 | 感覚運動クローズドループによる、Surprise（予測誤差）の動的収束。 |

## **3. 脳領域における数理的意図 (Mathematical Target)**

対話能力の獲得において、FERROの各脳モジュールは以下のように機能します。

1. **皮質（Cortex）の有糸分裂（Mitosis）と側抑制（Lateral Inhibition）**:
   新しい単語や文脈のインジェクションにより、局所自由エネルギー（FEP）が上昇します。皮質の有糸分裂（Mitosis）ロジック（`dynamic_cluster.rs`）は、親クラスター内のサブノード数（`concept_nodes`）が4つ以上あるときに動作する仕様であり、リプレイ時に動的に `ConceptNode` を追加・更新して有糸分裂を促すように修正されました。これにより、FEPの上昇に伴うクラスターの自律的分裂と自己組織化が正常に機能します。
2. **中脳（Midbrain）のマルチトークンルーティング ＆ 随伴発射相殺（Efference Cancellation）**:
   スペースで区切られたトークン（例：「いぬ が はしる」）は、中脳で個別にパースされ、それぞれのトークンアクター（「いぬ」「が」「はしる」）へとSurpriseが個別に分配されます。
   また、コア自身が「発話」した際、その運動コピー（`EfferenceCopy`）を耳アクターに送ることで、自身の発話による不要な驚愕度（Surprise）の上昇を相殺し、システムが自声に驚いてパニックを起こすのを防ぎます。
3. **海馬（Hippocampus）の短期バッファ ＆ 睡眠期（Sleep）の長期固定化**:
   覚醒中に起きた対話は海馬にエピソードバッファ（`episodic_buffer.csv`）として一時保存されます。FEPが減衰し睡眠期に遷移すると、海馬からこれらのエピソードがリプレイされ、皮質の結合強度がバックグラウンドで最適化されます。

## **4. 模擬チューター（Mock Tutor）による双方向学習システム**

人間が常時対話を入力し続けることは現実的ではないため、**Gemini API** を用いた「仮想対話相手（模擬チューター）」をホスト環境側に構築します。

```mermaid
graph TD
    subgraph Container [Docker Sandbox (ferro-core)]
        Core[FERRO Core]
        Core -- 運動出力 (vocal_output) --> Action[action/vocal_text.json]
        Sensory[stimulus/auditory.json] -- 感覚入力 --> Core
    end

    subgraph Host [Host Environment (ferro-shell / scripts)]
        Tutor[gemini_mock_tutor.py]
        Gemini[Gemini API]
        LocalDict[ローカルフォールバック辞書]
        
        Action -- 監視 (ポーリング) --> Tutor
        Tutor -- 対話文送信 --> Gemini
        Tutor -- オフライン時 --> LocalDict
        Gemini -- 応答生成 --> Tutor
        Tutor -- ひらがな分かち書き変換 --> Formatter[Formatter]
        Formatter -- アトミック書き込み --> Sensory
    end
    
    User[人間/飼育者] -- 直接入力 --> UserInput[user_input.json]
    UserInput -- 割り込み検知 --> Tutor
```

### **4.1. 特徴機能仕様**

1. **Gemini API 接続 ＆ ローカルフォールバック**:
   APIキーが存在する場合はGeminiの自然な対話能力を使用し、キーがない、もしくは通信エラーの場合は、高速なルールベース辞書（`handle_vocal_text` 拡張版）にフォールバックします。
2. **人間による割り込み（Human-in-the-Loop Interrupt）**:
   ダッシュボードから人間が入力し、`dashboard_chat_history.json` に最新のユーザー発話（`sender: "user"`）が記録された場合、その更新タイムスタンプを検知し、模擬チューターの自動応答を一定時間（例：120秒）ミュートします。これにより人間との対話が優先されます。
3. **人間会話エピソードの優先固定化（Surprise Boost）**:
   人間から直接入力された対話イベントには `is_human: true` の属性を疑似付与します。模擬チューターは、人間による入力に対するコアの反応を仲介する際、中脳または海馬に対して驚愕度（Surprise）を最大（`0.95`）にしてダンプを模倣します。これにより、睡眠期のリプレイにおいて人間との会話が強力に優先して再生され、少ない実会話から深く学習するアライメントが形成されます。

## **5. 大規模語彙の自己組織化とKVS自動移行（Storage Migration）**

FERROコアは、皮質内の累積ノード数が 5,000 件に到達したことを検知すると、ディスクI/Oスロットリングとトランザクション安全性を確保するため、分散された Sharded JSON（`/memory/knowledge_graph/clusters/*/*.json`）から単一KVS（`redb`）へ知識データベースを一括して自動移行します。

### **5.1. 移行シミュレーションの意図**
* **一意な語彙のインジェクション**:
  `generate_large_corpus.py` によって動的生成された 5,000 種類以上のユニークな日本語ひらがなトークンを含むコーパスをインジェクションすることで、睡眠期（Sleep）の海馬リプレイ時に、Cortexの動的生成プロセスが 5,000 個以上の独立した `ClusterNode` ファイルを自動的に創出します。
* **KVSマイグレーションのトリガー検知**:
  Cortexのクラスター数が 5,000 件に達した瞬間、コア内部の `StorageManager` が自動的に `redb` の書き出しトランザクションを完了し、以降の永続化I/OをKVSに一本化します。これにより、高密度環境下におけるデータベースの最適化性能を検証します。
